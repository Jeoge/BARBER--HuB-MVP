# Operations and Governance

この文書は、BARBER HUBの運営、審査、モデレーション、情報管理の方針をまとめます。

## 基本方針

- ユーザー投稿、店舗情報、広告、協賛情報を混同しない。
- 公開情報、申請情報、管理者確認情報を分けて扱う。
- 未確認情報を公式情報のように見せない。
- 審査や管理操作を一般ユーザーが直接実行できないようにする。
- 運用で必要な判断は、PRやdecision logに残す。
- 実数でない掲載数を表示しない。
- 情報取得元と利用条件を確認する。
- 個人情報や秘密情報を設計書へ保存しない。

## 店舗ディレクトリ運用

現在の店舗ディレクトリは、店舗検索と認証申請を扱います。

守るべき運用:

- 未認証店舗は、公式情報として見せすぎない。
- 認証申請は即時approved / verifiedにしない。
- 一般ユーザーが他人の店舗情報を編集できないようにする。
- 一般ユーザーが `verification_status` を変更できないようにする。
- 実在店舗を勝手に大量登録しない。
- テストデータは明らかにテスト用と分かる名称にし、削除方法を残す。
- 閉店、重複、名称変更、移転などの運用は今後の設計対象にする。

## 認証申請

認証申請では、店舗とユーザーの関係を確認する必要があります。

現時点の方針:

- 申請は `pending` から始まる。
- `verified` への変更は一般ユーザーに許可しない。
- 審査UIや管理画面は未実装として扱う。
- 申請履歴や申請者情報は、本人と運営以外に不用意に出さない。

## Back Room運用

Back Roomは会員限定コミュニティです。

守るべき運用:

- 未ログイン・未登録ユーザーには案内画面を出す。
- Back Roomプロフィールがないユーザーには参加設定導線を出す。
- 投稿やコメントは安心して使える場所を優先する。
- モデレーション、通報、非表示、審査画面は今後の設計対象として扱う。

## 投稿系の運用

- 投稿前の注意書きや安全確認を不用意に削除しない。
- 他人の投稿を編集・削除できないようにする。
- 投稿本文でHTMLやscriptを実行させない。
- 画像ファイルはサイズ、MIME type、安全なファイル名を確認する。
- エラー時にSupabaseの詳細エラーや秘密情報を画面へ出さない。

## 広告・協賛運用

- 広告、協賛、PRは通常コンテンツと誤認されないように扱う。
- 契約前の企業ロゴ、商品画像、キャンペーン情報を無断で使わない。
- 広告掲載問い合わせと通常投稿を分ける。
- 企業や学校の基本情報修正は原則無料として扱う。
- 公式情報と未認証情報を区別する。
- 「契約待ち」「未契約」など、運営の営業事情を一般画面へ表示しない。
- 掲載可否、表示順、料金、契約期間は今後の運用設計対象。

## 管理画面

現時点で本格的な管理画面は未実装として扱います。将来実装する場合は、BARBER HUB内の非公開 `/admin` 系領域を想定します。

守るべき方針:

- 一般会員には管理入口を表示しない。
- URLを知っていても権限なしでは入れない。
- 管理者操作はRLS、サーバー側処理、監査ログで守る。

将来必要になる管理対象:

- 店舗認証申請の審査
- 店舗データのCSV取込
- 閉店、重複、修正申請
- 投稿やコメントの通報対応
- 広告・協賛掲載管理
- 問い合わせ対応

## 要確認

- 管理者ロールの定義。
- 管理者操作ログの保存範囲。
- 審査担当者の権限分離。
- 削除、非表示、復元の運用ルール。
