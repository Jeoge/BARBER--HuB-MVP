# UI and Design

この文書は、BARBER HUBのUI変更時に守るべき原則をまとめます。

## 基本方針

- モバイルファーストで設計する。
- 375px、390px、430pxのスマホ幅を重視する。
- PC表示でも極端に大きく見えたり、余白が崩れたりしないようにする。
- 既存トップ画面の構成を不用意に変えない。
- 変更対象ではないカード、画像余白、背景色、下部ナビ、投稿ボタンを触らない。
- 説明するより、使えば分かる入口を優先する。

## 現在のトップ画面の主な構成

現在のトップ画面は、おおむね次の導線で構成されています。

- ヘッダー、ロゴ、スローガン
- ロゴ行右側の店舗検索ボタン
- カテゴリーナビゲーション
- EDITOR'S PICK
- 3 MIN NEWS
- SNAP
- 協賛・パートナー枠
- Back Roomカード
- Barber Directoryの掲載数・検索導線
- 記事、投稿、Q&A、求人、講習会などの横スクロール導線
- GUIDE / ルール系カード
- 下部ナビ
- 右下の投稿ボタン

この並びや見た目は、依頼の対象でない限り変更しません。

## UI変更の確認項目

- 375px、390px、430pxで横スクロールが出ない。
- PC幅でカードや画像が大きくなりすぎない。
- 文字がボタンやカードからはみ出さない。
- テキスト、画像、アイコン、ボタンが重ならない。
- absolute配置で不要なスクロールバーが出ない。
- ダークモードやOS設定で読みづらい色にならない。
- 下部ナビと右下投稿ボタンに干渉しない。
- クリック対象が小さくなりすぎない。
- Vercel Previewの実画面を重視する。
- ローカルだけのスクリーンショットを完成証明にしない。
- Preview Deploymentのcommit SHAとPRの最終commitを一致させる。

## 投稿者アイコン

- 投稿者を表す丸いプロフィールアイコンは、通常のフィード表示では40px前後を基準にする。
- 小型カードやコメントなど狭い表示では36px前後までを許容し、投稿者名、職種、日時を押しつぶさない。
- `avatar_url` を表示する共通コンポーネントでは、通常表示と小型表示のsize指定を使い、個別にサイズを乱立させない。
- アイコン画像は丸形、`object-cover`、fallback頭文字の中央表示を維持する。
- マイページのプロフィールメイン画像、`/profiles/[id]` のプロフィール画面、プロフィール編集画面、店舗プロフィールの大型アイコンへ波及させない。

## 色・カード・写真

- 基本は白背景、黒文字、淡いグレー、アクセントピンク。
- カードは既存のサイズ、角丸、枠線、影の方向性を保つ。
- ボタンやカードの既存サイズを不用意に変更しない。
- 写真は4:5前後を基本にし、高さ上限を設ける。
- 写真の左右余白やカード内余白は、対象PRでない限り変更しない。
- 広告カードを主役にしすぎない。
- 説明文を増やしすぎない。

## Snap画像表示

- Snapは画像なし、1枚、最大4枚まで表示できる。
- 複数画像は既存の画像表示領域内で横スワイプし、カードの幅・高さ・余白・リアクション位置を変えない。
- 2枚以上のときだけ控えめなドットを表示し、1枚では位置表示を出さない。
- トップ、Snap一覧、Snap詳細、マイページでは1枚目を従来と同じ位置・比率で表示する。
- Snap画像は遅延読み込みを基本にし、2枚目以降を一度にすべて先読みしない。

## 記事画像とEDITOR'S PICK

- 記事投稿画像は1枚までとし、画像なし記事でも既存表示が崩れないようにする。
- 記事画像は記事詳細、カテゴリー別一覧、マイページ、投稿者プロフィール、EDITOR'S PICKで同じ `image_url` を使う。
- トップのEDITOR'S PICKは、セクション位置、横スクロール、カード幅、写真アスペクト比、タグ位置、READ表示、余白を維持する。
- 運営選定記事が不足する場合は既存固定EDITOR'S PICKで補完し、運営選定記事が0件でもトップ画面の見た目を維持する。

## Thanks操作演出

- Snapと記事のThanks追加時は、ボタン周辺だけにピンクのキラキラが広がる短い操作演出を表示する。
- Thanks解除時、いいね、保存、コメントでは祝福演出を表示しない。
- 演出はレイアウト寸法を変えず、`prefers-reduced-motion` では移動・拡大を伴うburstを表示しない。

## 画像とpublic asset

- 画像を使う場合は、`public` 配下など本番ビルドで読める場所へ保存する。
- ローカルだけに存在する未追跡ファイルを参照しない。
- `.gitignore` により除外されていないことを確認する。
- パスの大文字・小文字はGitHub / Vercel上でも完全一致させる。
- Next/Imageを使う場合も、public配下のパスで本番ビルドに載ることを確認する。
- 画像の縦横比を変えない。
- 人物や商品など、見せるべき対象を切らない。

## Back Roomカード

- トップ画面のBack Roomカードは、既存のカードサイズ、余白、ボタン配置を壊さない。
- 右側に画像や線画を置く場合は、タイトル、説明、会員限定バッジ、ボタンに重ねない。
- カードから画像をはみ出させない。
- 本番とPreviewの表示を一致させるため、画像ファイルを必ずcommit / pushする。

## GUIDE / ルール系カード

- GUIDEカードはトップページだけに表示する。
- ルール本文ページ下部へGUIDEカードを出さない。
- 「法務」という言葉はトップ導線では使わない。
- GUIDEカードだけをダークグレーにする場合、トップ画面全体や他ページの背景は変更しない。
- 写真カード、記事カード、求人カード、協賛カードの画像余白を巻き込まない。

## 文言

- 画面内説明は短くする。
- 機能説明、注意書き、規約文を必要以上に増やさない。
- 未ログイン・未登録ユーザー向け案内では、次に何をすればよいかを明確にする。
- 規約やガイドライン本文は、UI変更PRで不用意に書き換えない。

## 禁止事項

- 依頼外のトップ画面再設計。
- 依頼外の背景色変更。
- 関係ない写真カードの余白変更。
- 画像をpublicに保存せず、ローカル参照のままPRを出すこと。
- Preview確認なしに、本番反映後の見た目を完成扱いにすること。
