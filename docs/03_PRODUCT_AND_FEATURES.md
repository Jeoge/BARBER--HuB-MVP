## 認証 / 会員登録

会員登録はSupabase Authを使い、ユーザーにメールアドレスとパスワードを1回入力してもらった後、確認メール経由でメール確認を完了します。

守るべき仕様:

- 登録直後は「確認メールを送信しました」と表示し、確認前にログイン入力へ誘導しない。
- 確認メールのリンクは `/auth/callback` で受け、Supabaseの `code` をセッションへ交換する。Supabaseメールテンプレートが `token_hash` 型の場合も同じcallbackで扱えるようにする。
- callbackでセッション交換に成功した場合は、直接ログインページや保護ページへ飛ばさず、専用のメール確認完了画面を表示する。
- 確認リンクが使用済み、期限切れ、二重クリック、またはパラメータ不足で処理できない場合も、エラー表示前に既存の有効セッションを確認する。既存セッションがあれば成功扱いで確認完了画面へ進める。
- メール確認完了画面では「メール確認が完了しました」と表示し、「BARBER HUBを開く」ボタンを1回押してもらう。
- ボタン押下時にServer Actionでログイン状態を再確認し、正常なら指定された安全な `next`、未指定ならトップ画面へ移動する。ボタンはpending中に「確認しています...」を表示し、disabledにして二重送信を防ぐ。
- callback失敗かつ有効セッションがない場合は、成功時と同じ「BARBER HUBを開く」文言を使わず、`/login?next=...` へ案内する。ログイン画面へ渡す `next` は `safeNextPath` 済みの値だけを使う。
- 正常なセッションがある場合は、メールアドレスとパスワードを再入力させない。
- メールアプリ内ブラウザなど別ブラウザで開いたためセッションが確認できない場合は、成功・失敗が分かる専用画面で案内し、会員登録のやり直しへ誘導しない。
- `next` はアプリ内の相対パスだけを許可し、外部URLや `//` から始まるURLへリダイレクトしない。

## 会員登録後の利用導線

- 会員登録時点では全員が個人アカウントとして始まる。
- Snap、記事、Back Room、Q&Aの利用に店舗登録は必要ない。
- プロフィールの登録区分は本人の職種や立場を表す表示用情報であり、店舗管理権限や求人掲載権限とは分けて扱う。
- 求人、事業承継、居抜き、備品売却、店舗情報管理などの店舗機能が必要なユーザーだけ、登録後にマイページからサロン機能を追加する。
- サロン機能は、店舗検索から既存店舗の管理申請、または見つからない場合の新規店舗登録申請へ進む。
- 店舗と個人アカウントが正式に紐づき、`barber_shops.verification_status = 'verified'` になった場合だけ、求人掲載や開業・承継掲載などの店舗機能を利用できる。
- プロフィールで「サロンオーナー」等を選んだだけでは、店舗管理権限や求人掲載権限を与えない。

## 店舗ディレクトリ

現在の店舗ディレクトリは、理容店舗の検索・閲覧・認証申請を扱います。

実装済みの範囲:

- 掲載店舗数の取得
- 店名検索
- 部分一致検索
- 都道府県・市区町村による地域検索
- 認証済み店舗のみの絞り込み
- 店舗詳細ページ
- 未認証店舗の「この店舗のオーナーですか？」導線
- ログインユーザーによる店舗登録申請
- 既存店舗への認証申請
- 管理者による店舗管理申請の一覧、承認、却下
- 認証済みオーナーによる自店舗情報編集

守るべき仕様:

- 一般ユーザーが自分以外の店舗を編集できない。
- 一般ユーザーが `verification_status` を `verified` に変更できない。
- 認証申請は即時verifiedにしない。
- 申請中店舗はマイページに「確認中」として表示し、認証済み店舗として扱わない。
- 店舗管理申請フォームでは、店舗のオーナー、代表者、または店舗情報を管理する正当な権限があることをチェック必須で確認する。
- 店舗管理申請の審査は管理者専用の `/admin/barber-shops/claims` で行い、一般ユーザーには入口を出さない。
- 口コミ、星評価、ランキングは実装しない。
- CSV取込管理画面は管理者専用の `/admin/barber-shops/import` で扱う。一般ユーザーには入口を出さない。
- メーカー、ディーラー、学校、組合の画面やデータを店舗PRへ混ぜない。
- 店舗カードは写真なしでも成立する小型1カラムを基本にする。
- 大量店舗を無限に全件表示せず、検索・地域絞り込み・ページネーションを使う。
- トップでは大量カードではなく、掲載店舗数と検索入口を見せる。

## Snap

Snapは、画像付き投稿とリアクションを扱う機能です。

実装済みの範囲:

- 画像なし / 1枚 / 最大4枚の画像アップロード
- 投稿本文
- 新規Snap投稿ではカテゴリーをユーザーに選択させず、保存上必要な内部既定値として既存カテゴリーの「日常」を使う
- Thanks
- コメント
- Snapコメントへのいいね
- 保存系
- 投稿前の安全確認

守るべき仕様:

- Snap画像は投稿前にブラウザ側で縮小・圧縮し、WebPを優先、必要時は圧縮JPEGへ変換する。
- 原寸画像や安全にデコードできないHEIC / HEIFはサーバーへ送信しない。
- 1投稿の圧縮後画像容量は最大4枚合計でおおむね4MB以内にする。
- 画像アップロードはサイズ、MIME type、ファイル名、圧縮後の画像シグネチャを検証する。
- 複数画像は選択順を表示順として扱う。
- 新しいSnap画像は `snap_images` を順序順に使い、既存の1枚Snapは `image_url` / `image_path` を1枚目としてフォールバック表示する。
- Snap表示では `storage_path` がある画像は現在のStorage URLを組み立てて表示し、空path、空URL、重複pathは表示配列に残さない。
- 自分の投稿へのThanksはポイント反映・公開競争に使わない。
- 投稿本文でHTMLやscriptを実行させない。
- 他人のSnapを編集・削除できないようRLSと操作条件を守る。

## プロフィール登録区分

- プロフィール設定の通常選択肢には「理美容アシスタント」「サロンスタッフ」を表示しない。
- 「アシスタント」は通常選択肢として維持する。
- 既存プロフィールに保存済みの「理美容アシスタント」「サロンスタッフ」は一括変更せず、既存値として表示・保存互換を維持する。

## Thanks / いいね / 保存 / フォロー

- SnapのThanksは実装済み。
- Snapと記事の公開画面では、Thanks、いいね、コメント、保存を既存の永続化基盤で扱う。
- 記事のいいね、Thanks、保存は実装済み。
- 保存系、フォロー系は既存機能として扱う。
- 自分自身のThanks・いいねは保存・加算対象外にする。
- Snapと記事の公開画面ではThanks数・いいね数を表示しない。
- Snapと記事の公開画面ではコメント数を表示し、0件も「コメント 0」と表示する。
- Snapと記事のThanks追加操作は共通のキラキラ演出で即時フィードバックし、解除時には表示しない。
- 公開中かつ未削除のSnapコメントは全ユーザーが閲覧でき、投稿はログインユーザーに限定する。
- 公開中かつ未削除のSnapコメントには、ログインユーザーがコメント単位でいいねできる。自分のコメントには押せない。
- Snapコメントいいねはコメントごとの公開数と、自分が押しているかだけを画面に返す。誰が押したかの一覧は公開しない。
- Thanks数・いいね数は公開競争に使わず、投稿者本人だけがマイページで確認する。
- 本人専用のマイページ指標と、公開ランキング表現を混同しない。
- 求人、開業・承継にはSnap/記事用のThanks・いいね・コメント・保存UIを追加しない。

## 記事

記事は、理容師本人の経験、経営、技術、集客、AI活用、講習会・コンクールレポートなどを共有する機能です。

実装済みの範囲:

- 画像なし記事投稿。
- 最大4枚の記事画像投稿。
- 講習会・講習会レポート・コンクールレポートで任意のYouTubeリンク設定。
- 記事詳細、カテゴリー別一覧、マイページ、投稿者プロフィールでの記事表示。
- 記事へのThanks、いいね、コメント、保存。
- 運営者だけが記事投稿時にEDITOR'S PICK掲載指定できる。
- 運営者だけがマイページの記事管理からEDITOR'S PICK掲載を解除できる。

守るべき仕様:

- 記事画像は投稿前にブラウザ側で縮小・圧縮し、WebPを優先、必要時は圧縮JPEGへ変換する。
- 原寸画像や安全にデコードできないHEIC / HEIFはサーバーへ送信しない。
- 記事画像は圧縮後1枚あたり最大約2MBにする。
- 1記事の圧縮後画像容量は最大4枚合計でおおむね4MB以内にする。
- サーバー側でもMIME type、容量、JPEG / WebPの画像シグネチャを再検証する。
- 投稿失敗、DB保存失敗、保存確認失敗時は、今回アップロードした記事画像を削除する。
- 新規記事画像はprivate bucketへ保存し、公開中かつ未削除の記事表示時だけサーバー側で短時間のsigned URLを発行する。
- image_pathがなく既存のimage_urlだけを持つ記事は、互換性のため従来のURL表示を維持する。
- 記事カード、トップ、カテゴリー別一覧、マイページ、投稿者プロフィール、EDITOR'S PICKでは代表画像として1枚目だけを使い、カードサイズや余白を変えない。
- 記事詳細では本文中の `[[article-image:1]]` から `[[article-image:4]]` のマーカー位置に対応画像を表示できる。本文に挿入済みの画像はトップ位置で重複表示せず、未挿入画像またはマーカーなし画像は従来のトップ画像位置に選択順で表示する。
- 直接動画アップロードは実装しない。対象カテゴリのYouTube URLはサーバー側でYouTubeドメインと動画IDを検証し、記事詳細ではiframe埋め込みではなく軽量な外部リンクとして表示する。
- YouTube URLは任意。動画権利確認は、対象カテゴリでYouTube URLが実際に入力された場合だけ必須にする。
- 一般ユーザーがFormDataを改ざんしてもEDITOR'S PICKに設定できないよう、Server Action側で運営者判定を行う。
- `editor_pick_at` がないPreview環境では、記事投稿とトップ表示がクラッシュしないようにし、固定EDITOR'S PICKへフォールバックする。

カテゴリー対応:

| 表示導線 | 記事カテゴリー |
| --- | --- |
| 経営 | `経営`, `独立`, `求人`, `経験記事` |
| 集客 | `集客` |
| AI | `AI活用`, `AI` |
| 技術 | `技術`, `講習会`, `講習会レポート`, `コンクールレポート` |
| 道具 | `道具`, `メーカー新商品`, `店販`, `開業準備` |

EDITOR'S PICK:

- 一般投稿者が自分で掲載を選ぶ仕様にしない。
- `BARBER_HUB_ADMIN_USER_IDS` を基本にし、互換用に `NEWS_REVIEW_ADMIN_USER_IDS` の運営者allowlistも利用できる。
- トップページでは `editor_pick_at` が設定された公開中・未削除記事を、最新の `editor_pick_at` 順で最大3件表示する。
- 運営選定記事が3件未満の場合は、既存固定EDITOR'S PICKで不足分を補完する。
- 実投稿記事のカードでは、投稿画像、記事タイトル、投稿者名、投稿者アイコン、プロフィールリンク、記事詳細リンクを使う。
- カテゴリーに応じた短い英語タグを表示し、カードサイズは変えない。

## アプリ内通知

実装済みの範囲:

- 自分のSnapへのThanks、いいね、コメント。
- 自分のSnapコメントへのいいね。
- 自分の記事へのThanks、いいね、コメント。
- 通知一覧、通知ごとの既読、すべて既読、Bottom Navigationの未読バッジ。

守るべき仕様:

- 通知はアプリ内通知のみ。Push通知、メール、SMS、Web Push、リアルタイム購読、通知設定画面はまだ実装しない。
- 自分自身の操作では通知を作らない。
- 通知文は操作したユーザーが分かる文章にする。
- 解除されたThanks、いいね、Snapコメントいいねの通知は残さない。
- 削除または非公開になったSnap/記事/コメントの通知は一覧で壊れたリンクにしない。
- 未読バッジは本人の未確認情報として小さく表示し、ランキングや人気競争の表現にしない。

## 記事 / Q&A / 求人 / 承継

これらは既存機能として扱い、店舗ディレクトリやUI調整PRで壊さない。

- 記事カード、求人カード、写真カードの画像余白は、対象PRでない限り変更しない。
- 求人、承継、Q&Aの投稿導線や保存処理を、関係ないPRで触らない。
- 各投稿画面の注意書きやチェック欄を不用意に変更しない。

トップページQ&A表示:

- トップページは `qa_questions` から公開可能な質問を最新順で最大3件取得して表示する。
- Q&A詳細へ遷移できる質問タイトル、カテゴリ、解決状態、公開コメント数を表示する。
- 固定ダミーQ&Aや、実データ0件時のダミーfallbackは表示しない。
- 実データが0件の場合は、Q&A見出しと既存のQ&A投稿導線だけを残す。
- 取得エラー時もダミー質問へ戻さず、トップ画面全体を壊さない範囲でQ&Aカードなしにする。

トップページCONTRIBUTION:

- CONTRIBUTIONは「経験を未来に残そう」「繋がれば　もっと面白い」を上部文言として使う。
- Thanks、投稿の積み重ね、マイページでの本人確認、バッジ、編集部セレクト、将来特典へつながる説明に絞る。
- カード内の不要な下部補足文は削除し、説明量と高さを抑える。
- 「将来の特典」カードの見出し、本文、下部文言、アイコンは維持する。

トップページ自動スクロールカード:

- Q&A下、メーカー新商品上にあった黒い自動スクロール記事カードは削除済み。
- メーカー新商品セクション、講習会、学生・求人の横スクロール導線は維持する。

## 3MIN NEWS自動化

実装済みの範囲:

- 公式または信頼できるRSS / APIからニュースメタデータを取得する。
- 理容師、サロン経営、技術、衛生、人材、お客様との会話との関連性を判定する。
- 関連性が低いニュースはAI下書き生成の対象外にする。
- AIはBARBER HUB向けの下書きを作成する。
- AIは仕事・経営、理容師本人、お客様との会話の3視点でタイトル候補を作成する。
- 下書きは `pending` の非公開状態で保存する。
- 運営者だけが `/news-review` でタイトル候補を比較し、最終タイトルを選択・修正してから確認、承認、非掲載、保留を行う。
- 運営者が `approved` にしたニュースだけをトップページの3MIN NEWSとニュース詳細へ表示する。
- 公開ニュースが不足または取得失敗した場合は、既存の固定ニュースをフォールバック表示する。

守るべき仕様:

- ニュースを自動公開しない。
- タイトル候補を選んだだけで公開せず、公開タイトルは最終的な `draft_title` とする。
- `title_candidates` と `primary_angle` は内部編集用で、一般ユーザー向け公開RPCには返さない。
- トップページの3MIN NEWSは最大4件表示を維持する。
- `pending`, `rejected`, 生成エラーあり、重複候補、不完全な下書きは公開しない。
- 元記事本文全体を保存・転載しない。
- HTML本文の無断スクレイピングをしない。
- 情報源にない数値、人物発言、断定をAIに作らせない。
- 既存の3MIN NEWS固定表示と `mockData.ts` は削除せず、フォールバックとして残す。

2026年7月時点の拡張方針:

- 若い男性理容師が毎日見たくなる入口として、3MIN NEWSを仕事・提案・会話の3方向へ広げる。
- 内部分類は `work`, `style`, `talk` を使い、公開UIへ内部スコアや内部理由は出さない。
- `work` は理容師の仕事、技術、経営、制度、衛生、安全、人材、AI、店舗運営に直接関係する情報を扱う。
- `style` はメンズファッション、メンズヘア、ヘアケア、頭皮ケア、メンズスキンケア、香水、身だしなみなど、施術提案・店販・男性客との会話に使える情報を扱う。
- `talk` は音楽、店内BGM、映画・ドラマ、エンタメ、主要スポーツ、若い男性の生活トレンドなど、接客中の自然な会話や店内体験づくりに使える情報を扱う。
- 芸能・エンタメは作品、活動、イベント、ファッション、ヘア、公式発表を中心とし、不倫、離婚、恋愛ゴシップ、私生活暴露、SNS炎上だけの記事は対象外にする。
- スポーツは主要大会、日本代表、日本人選手の大きな話題など、多くのお客様との会話に使える範囲に限定し、スポーツ速報サイト化しない。
- 医療的な発毛効果、美容効果、治療効果を断定する候補はAI前に除外または高リスク扱いにし、公式情報確認を前提にする。
- 音楽や店内BGMでは歌詞・音源・動画を保存または転載しない。店舗BGM利用条件は公式情報確認を促す。
- トップページの3MIN NEWSは最大4件のまま維持し、WORKだけ4件、TALKだけ4件のような偏りを避ける。
- 公開ニュースが不足する場合は、仕事、ヘア・美容、身だしなみ、会話ネタの固定fallbackで補完する。

2026年7月時点の追加運用:

- 有効取得元は、厚生労働省新着情報、IPA重要なセキュリティ情報、金融庁新着情報、総務省新着情報を基本とする。
- 流行系の追加取得元は、FASHIONSNAP、WWDJAPAN、音楽ナタリー、映画ナタリー、NHKニュース スポーツの正規RSSを使う。
- 取得元は `enabled`, `sourceGroup`, `priority`, `maxCandidatesPerRun` を持ち、問題がある取得元だけをコード上で停止できる。
- `sourceGroup` は `health_labor`, `business_money`, `digital_security`, `product_safety`, `fashion_beauty`, `music_entertainment`, `sports_talk` などを使い、1回の下書き生成で1分野だけに偏らないようにする。
- 1回のAI下書き候補は全取得元合計で最大6件とし、1分野あたり最大2件、同一取得元は原則最大2件までに抑える。
- AI下書き候補は、WORK最大3件、STYLE最大2件、TALK最大1件を目安にし、無制限に流行領域を増やさない。
- 候補化は新着順だけではなく、ルールベースの関連度、低品質行政告知の除外、AIによる3分価値判定を組み合わせる。
- AI判定では、理容師が朝・昼・夕方にBARBER HUBを開いて3分読む価値があるかを評価し、関連性、興味、緊急性、実用性、会話化しやすさを見ます。
- AIが使えない場合でも収集処理全体は止めず、ルールベース判定へfallbackする。
- 同一URL、正規化タイトル、同じ制度名・主要語句、直近7日以内の公開ニュースとの類似は候補から除外する。
- トップページの3MIN NEWSは4件表示のまま維持し、新しい公開ニュースが既存4件より低価値な場合に無理な総入れ替えを行う考え方にはしない。
- トップページでは、運営承認により公開状態になった `reviewed_at` から12時間以内の公開ニュースだけ、新聞アイコン上へ小さく `NEW` を表示する。

## CSV店舗取込管理

実装済みの範囲:

- 管理者専用CSV選択画面。
- CSV内容プレビュー。
- 必須項目チェック。
- 完全一致重複と類似候補の表示。
- 登録実行と登録結果表示。
- 取込バッチと取込行の履歴保存。

守るべき仕様:

- CSVデータ本体をmigrationやseedに含めない。
- 店名、都道府県、市区町村、掲載元は必須にする。
- 住所と電話番号は空欄でも取込可能にする。
- 電話番号はtextとして保存し、先頭0を落とさない。
- CSVの「未認証」は `unverified` として保存し、認証済みのように見せない。
- 久留米支部組合員名簿由来のデータを「久留米市全店舗」と断定しない。
