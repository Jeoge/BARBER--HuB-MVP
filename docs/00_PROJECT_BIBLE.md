# BARBER HUB Project Bible

BARBER HUBは、理容師・サロン・理容学生・学校・メーカー・ディーラー・組合・求人関係者等をつなぐ、理美容業界向けの中立的な業界プラットフォームです。

単なるSNS、一般客向け口コミサイト、広告一覧サイトではありません。

この文書は、BARBER HUBの最上位方針です。細かなUI値やテーブル定義ではなく、長期的に守る判断基準を記載します。

## 中心思想

- One Success. Shared Success.
- 成功や経験を共有する。
- まず自分のために開く価値を作る。
- 経験共有、Thanks、信頼へつなげる。
- 数字競争、フォロワー競争を過度に持ち込まない。
- 業界内の情報格差や世代間の分断を減らす。
- 組合・企業・学校のどこか一者へ主導権を渡さず、中立性を保つ。
- 理容師が毎朝最初に開く場所を目指す。
- 特に若い理容師が、仕事にも接客中の会話にも使える情報を毎日確認できる場所を目指す。

## 絶対に守ること

- 店舗口コミを実装しない。
- 星評価を実装しない。
- 店舗ランキングを実装しない。
- フォロワー数を競争の中心にしない。
- 未認証企業を提携先・スポンサーのように見せない。
- 有料広告を編集部記事に偽装しない。
- PR、広告、Sponsored、協賛、Partner等を明示する。
- 他社サイトの口コミ、紹介文、写真等を無断転載しない。
- Googleマップ等を無断スクレイピングしない。
- 現在のブランド世界観を、機能追加のたびに全面変更しない。
- mainへ直接commit / push / mergeしない。
- 本番DBで `supabase db reset` を実行しない。
- Secrets、DBパスワード、接続文字列、tokenをコード・docs・ログへ出さない。
- `service_role` keyをクライアントコードへ入れない。

## 店舗ディレクトリの基本方針

- Google Business Profile型の考え方を採用する。
- 運営が最低限の公開店舗情報を先に掲載できる。
- 店舗名と地域等は検索可能な公開情報として扱う。
- 未認証店舗は写真・URL・口コミなしでよい。
- オーナーが後から認証申請し、認証後に情報を充実できる。
- 掲載店舗数は、実際に公開検索可能な店舗だけを数える。
- 店舗情報とユーザーアカウントは別管理にする。
- 登録されていない店舗は新規登録できる。
- 閉店、誤情報、掲載拒否、修正申請へ対応できる設計にする。
- 一般ユーザーが他人の店舗や `verification_status` を変更できないことを守る。

ただし、無料の基本掲載、公式認証、有料広告・告知を明確に分けます。今回の店舗ディレクトリPRに、これらの画面やデータを混ぜません。

## UIの基本方針

- スマートフォンファースト。
- 375px / 390px / 430pxで確認する。
- 既存トップ画面の見た目を壊さず、小さなPRで進める。
- ロゴ、スローガン、カテゴリーメニュー、EDITOR'S PICK、3 MIN NEWS、SNAP、Back Room、Barber Directory、下部ナビ、投稿ボタンを不用意に変えない。
- 写真カード、記事カード、求人カード、協賛カードの画像余白を関係ないPRで変更しない。
- 実画像やpublic assetを使う場合は、必ずリポジトリへcommitし、本番ビルドで参照できるパスにする。

## DB / Supabaseの基本方針

- 本物のSupabase永続化を基本とし、ダミー表示だけで完成扱いにしない。
- RLS、権限、セキュリティを後回しにしない。
- クライアントは公開可能なanon / publishable keyだけを使う。
- migrationは正式なschema変更手段として扱う。
- 適用済みmigrationを編集せず、修正は新しいmigrationで追加する。
- seedとmigrationを分離する。
- 本番seedを自動投入しない。
- 本番migrationはGitHub `production` Environmentの承認後に適用する。
- 詳細は [SUPABASE_DEPLOYMENT.md](SUPABASE_DEPLOYMENT.md) を正とする。

## 開発の原則

- スマートフォンファースト。
- 現在の見た目を壊さず、小さなPRで進める。
- 本物のSupabase永続化を基本とし、ダミー表示だけで完成扱いにしない。
- セキュリティ、RLS、権限を後回しにしない。
- ユーザーが原則マージとProduction承認だけで進められる環境を維持する。
- 実装済み事実、正式決定、検討中案を混同しない。

## 変更時の確認

- `pnpm build` を実行する。
- 既存のlint / typecheck / testコマンドがある場合は、必要に応じて実行する。
- UI変更ではVercel Previewや実ブラウザでスマホ幅とPC幅を確認する。
- DB変更ではmigration check、dry-run、RLS、RPC、権限を確認する。
- PR本文には、実施した確認と未確認項目を分けて書く。
